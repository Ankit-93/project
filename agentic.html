Agentic AI Based ChatBot for Healthcare Account Plan Analysis

Introduction

"One of my most impactful projects was developing an Agentic Bot Framework that streamlined healthcare account plan data analysis using LlamaIndex. The goal was to create an intelligent system comprised of specialized agents to analyze Elevance Health data through structured workflows, enabling comprehensive healthcare data analysis and natural language query processing."

Problem Statement

Navigating healthcare account plans is a complex and time-intensive process. These plans contain vast amounts of fragmented data—ranging from benefit names to cost shares across different network tiers—making manual validation and insight extraction both inefficient and error-prone.

During the annual renewal of commercial health benefit plans, Enrollment and Benefit Administration (EBA) professionals and Sales Agents must specify benefits and their cost shares for each network tier. This often requires searching across historical benefit data, manually locating relevant details within extensive documents—a process that slows down renewals and impacts efficiency.

To streamline this, we have developed an LLM-powered conversational AI assistant within Automated Benefit Integration (ABI). This intelligent, agent-driven framework:

  • Enhances efficiency by simplifying plan information access.
  • Reduces errors with real-time data validation.
  • Expedites renewals by minimizing manual research.
  • Improves collaboration through seamless integration with tools like Jira for task management.
  • Elevates the customer experience by accelerating speed to market and improving service responsiveness.

By leveraging AI-driven automation, our solution transforms benefit administration, optimizing workflows, ensuring data accuracy, and empowering teams with intelligent insights.

Architecture Overview:

Key Components & Their Roles

1. User Interface (UI)

Role: The front-end layer where users (e.g., Client Managers, EBAs) interact with the system.

Function: Submits queries (e.g., benefit plan questions) and receives responses or documents.

2. Agentic Controller

Role: Central orchestrator managing user requests and routing them to appropriate tools.

Function:

  • Maintains Session Context (tracks conversation history and user intent).
  • Coordinates with Benefit Coverage Tool (for benefit plan configurations), Plan Level Tool Plan/Group Level Info, and Engage Advisor (support ticketing).

3. Benefits Tool

Role: Core module for handling benefit-related operations.

Function:

  • Uses RAG (Retrieval-Augmented Generation) to fetch data from databases (e.g., Oracle DB, Post-Gross Vector DB).
  • Integrates LLM Call (e.g., GPT-4) to generate human-like responses to user queries.
  • Mostly replies in a single line (Example: "Air Ambulance is covered in most of the plans").

4. Statistics Tool

Role: Core module for statistics-related queries.

Function:

  • Accesses Oracle data through Procedure codes.
  • Formats the data and sends it to LLM for proper formatting as instructed by the prompt.

5. Costshare Tool

Role: Core module to share cost shares or plan-level information.

Function:

  • Accesses Oracle data through Procedure codes.
  • Formats the data and sends it to LLM for proper formatting as instructed by the prompt.
  • LLM responds with plan names or cost shares as a string. Using an output parser (Pydantic Output Parsers), we convert it to a list.

6. Databases

  • Oracle DB: Stores structured benefit data (e.g., plan details, cost shares).
  • Post-Gross Vector DB: Likely a specialized database for post-calculation or actuarial data.

7. Jira Creation

Role: Presumed to be an API Gateway (e.g., Client API) for external integrations.

Function: Connects to systems like JIRA (for Engage Advisor tickets) or FOBS (for plan updates).

Basic Workflow Example

User Query: What’s the cost share for air ambulance?

Agentic Controller: checks the Session Context to understand intent.

Query is routed to the Benefits Tool, which:

RAG to retrieve relevant data from Vector DB.

Generates a response via LLM Call, ensuring accuracy and natural language formatting.

User : Not satisfied with the response

User reports: Jira Creation compiles data into a formatted document.

Complex issues: Engage Advisor creates a ticket via JIRA API to involve SMEs.

Key Information

All interactions with the LLM are query-specific, meaning they do not retain previous context or chat history. To maintain continuity, we explicitly track context within Agentic AI.
Once the query is completed, we store the session data in Redis for efficient retrieval and state management.

AI Agentic Framework

How do we achieve this? We create a session where all user queries and metadata are stored.

Session Data Structure

Below is an example session:

            {
                sessionId:     16-digit UUID assigned to an individual user for a session,
                groupId:       metadata – identifier- group,
                benefitName:   Benefit extracted from user query,
                benefitOption: Cost share extracted from user query,
                planName:      Plan names extracted from user query
            }

User Logs In:

Response: Greet Message

Backend: session id is allocated to user

User : What’s the cost share for air ambulance?

The system performs basic entity extraction for groupId, groupName, benefitOption, and planName.

OpenAI agent is invoked with session data.

The AI Agent determines which tool should handle the query based on system prompts.

The tool validates metadata. If a group ID is missing, it prompts the user for input.

The tool retrieves benefit names using BM25 encoding for similarity search.

If multiple benefit names match, the user is prompted to select the correct one.

Data Fetching Mechanism

  • If metadata is complete, data is fetched from PostgreSQL vector DB using similarity search.
  • To avoid slow responses with large datasets, an alternative Oracle tabular database is used.
  • The data is preprocessed and formatted before being sent to LLM for insights.

Handling Plan-Level Queries

  • If a user asks for plan-level information, Oracle DB is used to retrieve group IDs and plans.
  • The response is formatted and passed through an output parser.

Handling Dissatisfaction

If the user is dissatisfied with the response:

  • A "thumbs down" option allows them to raise a dissatisfaction report.
  • If approved, the Engage Advisor creates a JIRA ticket for SMEs.

Leveraging AI Agents

AI Agents are transforming our solutions by enhancing efficiency, improving response time, and providing intelligent query handling.

Pros of AI Framework

  • Understands human language very well.
  • Fast Responses: Sometimes answers user queries from CACHE without querying tools, improving speed.
  • Handles Out-of-Scope Queries: Even if a question is outside its predefined scope, the bot provides intelligent responses, enhancing user experience.

Cons of AI Framework

  • Inconsistent Responses: LLMs may not always provide the same answer for identical queries.

Example of Inconsistent Responses

User Query: Is air ambulance covered?
Response 1: Air Ambulance is Not Covered in few of the plans.
Response 2: Air Ambulance is Covered in most of the plans.

Issue: While both responses appear similar, they have different meanings since "Covered" and "Not Covered" are distinct administrative types of coverage.
